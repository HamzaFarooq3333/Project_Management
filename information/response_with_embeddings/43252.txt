### Phases and Activities

The project will follow the PRINCE2 process model, structured into distinct management stages, ensuring continuous review and control by the Project Board.

#### I. Starting Up a Project (SU)
This phase focuses on ensuring there is a sound basis for the project before significant resources are committed.
i.  **Appoint Executive & Project Manager:** Establish the Project Board's Executive and appoint the Project Manager. The Executive is responsible for obtaining resources and executive decisions to enable success [Ref 4].
ii. **Capture Project Mandate & Create Project Brief:** Understand the project's purpose and objectives [Ref 4]. Develop an initial Project Brief, outlining the high-level scope and context [Ref 10].
iii. **Design and Appoint Project Management Team:** Define and appoint other key project roles necessary for governance and management.
iv. **Outline Stage Plan for Initiation:** Create a plan for the subsequent Initiation Stage.

#### II. Directing a Project (DP)
This ongoing process ensures that the Project Board has the ability to direct and control the project throughout its lifecycle.
i.  **Authorize Initiation:** Review the Project Brief and Stage Plan for Initiation; approve the project to proceed to the Initiation Stage.
ii. **Authorize the Project:** Review the Project Initiation Documentation (PID) after the Initiation Stage and authorize the main project work to commence.
iii. **Authorize a Stage or Exception Plan:** Review Stage End Reports or Exception Reports and approve subsequent stages or management of exceptions. This is a critical project governance point [Ref 9, Ref 12].
iv. **Give Ad Hoc Direction:** Provide guidance and decisions as required during the project's execution.
v. **Authorize Project Closure:** Approve the formal closure of the project upon completion.

#### III. Initiating a Project (IP)
This phase involves detailed planning and establishment of the project's foundation.
i.  **Plan the Project:** Develop a comprehensive project plan, including product scope and project scope definition, estimating, scheduling, and budget [Ref 6].
ii. **Refine Business Case:** Detail the justification for the project, confirming its value focus [Ref 8].
iii. **Establish Project Controls:** Define how the project will be monitored and controlled (e.g., reporting, issue management).
iv. **Create Project Initiation Documentation (PID):** Compile all planning documentation into the PID, covering development approach and life cycle [Ref 2], communication plan, procurement, change management, and measurement strategies [Ref 6].
v. **Set Up Project Files:** Organize and establish the project's documentation system.
vi. **Assemble Project Management Team:** Finalize the project team composition [Ref 6].

#### IV. Controlling a Stage (CS)
This phase covers the day-to-day management of a stage, where the Project Manager oversees execution and ensures products are developed as planned.
i.  **Authorize a Work Package:** Assign work packages to Team Managers [Ref 5].
ii. **Review Work Package Status:** Monitor progress and performance of assigned work.
iii. **Capture and Examine Issues & Risks:** Identify and analyze project issues and threats [Ref 5, Ref 14]. Develop and optimize risk responses [Ref 8].
iv. **Report Progress:** Regularly update the Project Board on stage progress.
v. **Take Corrective Action:** Implement necessary adjustments to keep the stage on track.

#### V. Managing Product Delivery (MP)
This phase focuses on the Team Manager's role in delivering the specialist products (research deliverables) agreed upon with the Project Manager.
i.  **Accept, Execute, and Deliver a Work Package:** Team Managers receive work packages, execute the activities, and deliver the specified products [Ref 5, Ref 12].
ii. **Undertake Research Activities:** Conduct the core research tasks, including data collection, analysis, literature reviews, and report generation, in line with the project's deliverables.
iii. **Liaise with Project Manager:** Maintain communication to report progress and raise any issues.

#### VI. Managing a Stage Boundary (SB)
This phase occurs at the end of each management stage, allowing the Project Board to review the stage, assess viability, and make decisions for the next stage.
i.  **Report Stage End:** Prepare a Stage End Report summarizing performance, issues, and lessons learned.
ii. **Update Project Plan & Business Case:** Revise the overall project plan and business case based on actual progress and any changes.
iii. **Produce Next Stage Plan:** Develop a detailed plan for the upcoming management stage.
iv. **Seek Project Board Authorization:** Present the Stage End Report and Next Stage Plan to the Project Board for review and authorization to proceed [Ref 9].

#### VII. Closing a Project (CP)
This final phase ensures the project is properly closed out, formally confirming achievement of objectives and preparing for benefits realization.
i.  **Prepare Planned Closure/Early Closure:** Execute activities for orderly project termination.
ii. **Hand Over Products:** Ensure all final products (e.g., research findings, reports, recommendations) are formally accepted and transferred to the appropriate operational entity.
iii. **Evaluate Project:** Assess the project's performance against its objectives and value [Ref 8].
iv. **Recommend Project Closure:** Present the End Project Report and Lessons Learned Report to the Project Board and recommend formal project closure.
v. **Archiving:** Archive project documentation and records as per organizational standards [Ref 8].

### Roles

The following key roles are essential for the effective management of this PRINCE2-driven research project:

1.  **Project Board:** Responsible for overall project direction and control [Ref 9, Ref 12]. Comprised of:
    *   **Executive:** Accountable for the project, ensuring it delivers value, obtains resources, and makes executive decisions [Ref 4].
    *   **Senior User:** Represents the interests of those who will use the project's products or be affected by them, ensuring the solution meets user needs.
    *   **Senior Supplier:** Represents the interests of those who provide the resources and expertise to produce the specialist products (e.g., research team members, external consultants).
2.  **Project Manager:** Responsible for the day-to-day management of the project within the boundaries set by the Project Board. This involves coordinating activities to direct and control the accomplishment of agreed objectives [Ref 4].
3.  **Team Manager:** (Optional, but recommended for medium projects) Responsible for managing a team to produce the required specialist products. This involves managing work packages [Ref 5].
4.  **Project Assurance:** An independent role to assure the Project Board that the project is being conducted correctly.
5.  **Project Support:** Provides administrative and project management support (e.g., planning, risk and issue logging, configuration management).
6.  **Stakeholders:** Individuals, groups, or organizations that have interests in, or can affect, be affected by, or perceive itself to be affected by, any aspect of a project [Ref 4, Ref 5]. Effective engagement with stakeholders is a key principle [Ref 8].

### Decision Gates

Decision gates are crucial control points where the Project Board assesses the project's continued viability and authorizes progression. These gates are aligned with PRINCE2's stage-based approach and governance principles [Ref 9, Ref 12].

1.  **Authorization to Initiate (End of SU):** The Project Board reviews the Project Brief and initial Stage Plan. Decision: Proceed with detailed planning (Initiation Stage) or stop.
2.  **Authorization of the Project (End of IP):** The Project Board reviews the complete Project Initiation Documentation (PID) and the Stage Plan for the first delivery stage. Decision: Authorize the project to proceed with delivery stages or stop. This decision is critical as it commits significant resources.
3.  **Stage Boundary Review (End of CS / Beginning of SB):** At the end of each delivery stage, the Project Board reviews the Stage End Report, assesses project progress, updated business case, and risks [Ref 14]. Decision: Authorize the next stage, request an exception plan, or stop the project. This gate occurs multiple times throughout the project lifecycle [Ref 10].
4.  **Final Project Closure Authorization (End of CP):** The Project Board reviews the End Project Report and Lessons Learned Report. Decision: Formally close the project, confirming all objectives have been achieved and products handed over [Ref 4].
